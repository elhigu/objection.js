parseFieldExpression (expression, extractAsText) → {Array}

Field expression how to refer certain nested field inside jsonb column.

Table.jsonColumn:obj[key.with.dots][0].key.0.me[0] refers to obj = { "key.with.dots" : [{"key" : { "0": { me : [ "I was referred" ] }}}]

Since PostgreSql #>{field,0,field2,...} operator does not make difference if reference is string or a number, one can actually use also jsonArray.0 notation to refer index of an array. Like wise one can use object[123] notation to refer key of an object { "123" : null }.

Parameters:
Name Type Description
expression String
extractAsText Boolean

Return text instead of jsonb object (useful for type casting).
Returns:

Array of referred path, where first item is table/column.

transaction () → {Promise}

Starts a transaction.

Give the the model classes you want to use in the transaction as arguments to this function. The model classes are bound to a newly created transaction and passed to the callback. All queries created using the bound model classes or any result acquired through them take part in the same transaction.

You must return a promise from the callback. If this promise is fulfilled the transaction is committed. If the promise is rejected the transaction is rolled back.

Examples:

objection.transaction(Person, Animal, function (Person, Animal) {

 return Person
   .query()
   .insert({firstName: 'Jennifer', lastName: 'Lawrence'})
   .then(function () {
     return Animal.query().insert({name: 'Scrappy'});
   });

}).then(function (scrappy) {
  console.log('Jennifer and Scrappy were successfully inserted');
}).catch(function (err) {
  console.log('Something went wrong. Neither Jennifer nor Scrappy were inserted');
});

Related model classes are automatically bound to the same transaction. So if you use Animal implicitly through Person's relations you don't have to bind Animal explicitly. The following example clarifies this:

objection.transaction(Person, function (Person) {

 return Person
   .query()
   .insert({firstName: 'Jennifer', lastName: 'Lawrence'})
   .then(function (jennifer) {
     // This insert takes part in the transaction even though we didn't explicitly
     // bind the `Animal` model class.
     return jennifer.$relatedQuery('pets').insert({name: 'Scrappy'});
   });

}).then(function (scrappy) {
  console.log('Jennifer and Scrappy were successfully inserted');
}).catch(function (err) {
  console.log('Something went wrong. Neither Jennifer nor Scrappy were inserted');
});

Inside the callback this is the knex transaction object. So if you need to create knex queries you can do this:

objection.transaction(Person, function (Person) {
 var knex = this;

 return Person
   .query()
   .insert({firstName: 'Jennifer', lastName: 'Lawrence'})
   .then(function (jennifer) {
     return knex.insert({name: 'Scrappy'}}.into('Animal');
   });

}).then(function () {
  console.log('Jennifer and Scrappy were successfully inserted');
}).catch(function (err) {
  console.log('Something went wrong. Neither Jennifer nor Scrappy were inserted');
});

whereJsonbRefOnLeftJsonbValOrRefOnRight (builder, fieldExpression, operator, jsonObjectOrFieldExpression, queryPrefix) → {QueryBuilder}

Where jsonb reference on left hand side is compared to jsonb value or reference on the right hand side.

Converts left and right hand values to PostgreSQL acceptable format and add user chosen operator between left and right hand expressions.

Type Definitions

FieldExpression

Json field expression to refer to jsonb columns or keys / objects inside columns.

e.g. Person.jsonColumnName:details.names[1] would refer to column Person.jsonColumnName which has { details: { names: ['First', 'Second', 'Last'] } } object stored in it.

ModelOptions

Type:

Name Type Description
patch Boolean

If true the json is treated as a patch and the required field of the json schema is ignored in the validation. This allows us to create models with a subset of required properties for patch operations.
skipValidation Boolean

If true the json schema validation is skipped.

RelationJoin

An object literal that describes how two tables are related to one another. For example:

{
  from: 'Animal.ownerId',
  to: 'Person.id'
}

or in the case of a many-to-many relation:

{
  from: 'Person.id',
  through: {
    from: 'Person_Movie.actorId',
    to: 'Person_Movie.movieId'
  },
  to: 'Movie.id'
}
Type:
  • Object
Properties:
Name Type Description
from String

The relation column in the owner table. Must be given with the table name. For example Person.id. Note that neither this nor to need to be foreign keys or primary keys. You can join any column to any column.
to String

The relation column in the related table. Must be given with the table name. For example Movie.id. Note that neither this nor from need to be foreign keys or primary keys. You can join any column to any column.
through Object

Describes the join table if the models are related through one.

Properties
Name Type Description
from String

The column that is joined to from property of the RelationJoin. For example Person_Movie.actorId where Person_Movie is the join table.
to String

The column that is joined to to property of the RelationJoin. For example Person_Movie.movieId where Person_Movie is the join table.
